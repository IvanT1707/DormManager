# Вимоги курсового проєкту DormManager

Цей файл фіксує спеціальну частину завдання як основний орієнтир для
подальшої розробки. Схеми з лабораторних робіт залишаються базовою моделлю,
але структура даних та функції можуть розширюватися, якщо це покращує
веб-застосунок і не суперечить вимогам.

## Спеціальна частина завдання

1. В огляді літератури проаналізувати відкриті джерела щодо управління
   діяльністю студентських гуртожитків та можливостей сучасних програм для
   автоматизації такого управління.
2. Розробити та реалізувати веб-застосунок із можливостями формування,
   редагування, збереження та доступу через інтерфейс користувача студента,
   персоналу гуртожитку й адміністратора до інформаційної бази системи,
   виконання операцій над даними, формування та роздруку звітів за запитами
   користувача, аналізу та візуалізації даних.
3. Для проектування та моделювання використати методологію Kanban та
   інструменти Draw.io.
4. Для програмних рішень використати JavaScript та СУБД PostgreSQL.

## Матриця відповідності

| Вимога | Поточна реалізація | Статус / наступне посилення |
| ------ | ------------------ | ---------------------------- |
| JavaScript та PostgreSQL | React/Vite frontend, Node.js/Express backend, PostgreSQL schema and queries | Реалізовано |
| Ролі користувачів | `student`, `commandant`, `maintenance_staff`, `administrator`; Firebase Auth і RBAC | Реалізовано; потрібно створити демонстраційні акаунти ролей |
| Доступ до інформаційної бази через UI | Студентські заявки й оплати; керування кімнатами, тарифами, поселеннями й ролями; черга ремонтів персоналу | Реалізовано для основного сценарію |
| Реєстраційні дані студента | Підтвердження пароля, відновлення пароля через Firebase, вибір реальних інститутів/освітніх програм Львівської політехніки | Реалізовано для каталогу ІКНІ, ІКТА та ІАРД з ручним доповненням |
| Формування, редагування та збереження даних | CRUD API для гуртожитків, кімнат, користувачів, послуг, поселень, заявок і транзакцій | Реалізовано |
| Структура житлового фонду | Поверхи, блоки, кімнати у блоках і пакетне створення кімнат за конфігурацією гуртожитку | Реалізовано; підтримує реальну схему гуртожитку №11 |
| Нумерація кімнат | Блочне планування з назвами `201а`, `201б`; звичайне планування без блоків із назвами `201`, `202` | Реалізовано в міграції `010_room_naming_layout_mode.sql`, API та формі адміністратора |
| Процес поселення НУЛП | Заяви без вибору кімнати студентом, призначення місця комендантом, продовження, переселення, виселення й контроль оформлення | Реалізовано за аудитом `LPNU_RESIDENCE_PROCESS_REVIEW.md` |
| Широкий предметний аудит Студмістечка НУЛП | Проаналізовано тарифи, місткість, пільги, перепустки, відвідувачів, майно, порушення, ремонти та ролі | Оформлено в `LPNU_FULL_DOMAIN_AUDIT.md`; визначено етапи P0-P2 |
| P1/P2 розширення | Персональні сповіщення, автоматичні нарахування, інтернет кімнати, scoped персонал, ремонтні коментарі та дисциплінарні бали | Реалізовано в міграціях `005`-`013`, API та React-панелях; опис у `P1_P2_ARCHITECTURE_PROPOSAL.md` |
| Звіти за запитом та друк | Друкований звіт про зайнятість, заявки й платежі з фільтрами за гуртожитком та періодом | Реалізовано для основного сценарію; можна розширити експортом |
| Аналіз та візуалізація | Картки показників, зайнятість гуртожитків, статуси заявок і місячні платежі з фільтрами | Реалізовано для управлінської панелі |
| Огляд літератури й відкритих аналогів | Додано аудит офіційного процесу НУЛП у `LPNU_RESIDENCE_PROCESS_REVIEW.md` | Доповнити порівнянням програм-аналогів у пояснювальній записці |
| Kanban | Не є кодом застосунку | Оформити дошку етапів та додати підтвердження процесу до звіту |
| Draw.io | Не є кодом застосунку | Підготувати діаграми процесів, даних і архітектури |

## Ролі та сценарії демонстрації

| Роль | Сценарій |
| ---- | -------- |
| Студент | Реєстрація, перегляд поселення, подання заявок, оплата нарахувань і перегляд дисциплінарного обліку |
| Комендант | Робота з кімнатами свого гуртожитку, обробка заяв, оформлення поселень і дисциплінарних записів, друк звітів |
| Обслуговуючий персонал | Черга заявок за спеціалізацією, статус робіт та публічні/внутрішні коментарі |
| Адміністратор | Тарифи, профілі/ролі, призначення персоналу, правила дисципліни та операції коменданта |

## Заплановані функціональні доробки

1. `P0`: додати паспортну місткість гуртожитків і не використовувати
   приблизну структуру гуртожитку №11 як офіційні `864` місця без уточнення.
2. `P0`: доповнити реалізовані періодичні нарахування пільгами та окремим
   звітом заборгованості.
3. `P1`: додати кампанії перепоселення за навчальним роком, категорії
   мешканців, погодження інститутів, перепустки, майно й обхідні листи.
4. `P1`: доповнити реалізований scoped модуль ремонтів пріоритетами й
   призначенням конкретного виконавця всередині спеціалізації.
5. `P2`: додати журнал відвідувачів, рішення комісії, оголошення та
   додаткову управлінську аналітику.
6. Підготувати матеріали курсового: порівняння програм-аналогів,
   Kanban-дошку та Draw.io-діаграми.

Детальний план міграцій та архітектури для пунктів P1/P2 наведено у
`P1_P2_ARCHITECTURE_PROPOSAL.md`.
Поточні зауваження ручного тестування та наступні зміни інтерфейсу й
предметної моделі накопичуються у `TESTING_FEEDBACK_BACKLOG.md`.

## Принципи розширення моделі

- Базові сутності лабораторних (`dorm`, `room`, `users`, `service`,
  `residence`, `application`, `transactions`) зберігаються як ядро системи.
- Нові таблиці або поля додаються через окремі міграції, якщо вони
  підтримують звітність, аналітику або реальні рольові процеси.
- Оплата для курсової роботи залишається імітованою; реальні платіжні дані не
  обробляються.
- Firebase використовується для автентифікації, а PostgreSQL - для предметних
  даних і ролей DormManager.
